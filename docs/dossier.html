Volume I · Section 1

Executive Summary & Value Proposition

The Dhurta Ecosystem is a sovereign, privacy-native browser platform engineered for individuals and organisations who regard digital surveillance as an unacceptable operating condition. Unlike conventional browsers that treat privacy as a configurable add-on, Dhurta embeds privacy guarantees at the architectural layer — zero telemetry, zero cloud dependency, and zero negotiation with the attention economy.

The ecosystem addresses three compounding market failures. First, the systematic erosion of IP anonymity through persistent browser fingerprinting, WebRTC protocol leaks, and third-party tracker networks that survive even cookie deletion. Second, the absence of a secure, zero-infrastructure communication channel for sensitive peer-to-peer collaboration. Third, the usability gap that has historically made privacy tooling the exclusive domain of technically specialised users.

Dhurta resolves these failures through a vertically integrated product suite: a primary Sovereign Browser with Chakra Shield privacy mode and one-tap Ghost Mode Tor routing; Dhurta Connect, a zero-server P2P encrypted messaging and file-transfer layer; and Dhurta Setu, a curated web index and bridge to the open information commons — all unified within a single Electron-native desktop application.

The value proposition is singular: the user retains complete sovereignty over every byte of data generated in the course of their digital activity, with no exceptions, no opt-outs required, and no external dependencies that can be compelled, subpoenaed, or compromised.

Market Problem Solved
IP Identity Exposure
Every unprotected browser session broadcasts a stable IP address to every server contacted. Dhurta's VPN integration and Tor-backed Ghost Mode eliminate this attack vector at the network layer.
Market Problem Solved
Browser Fingerprinting
Screen dimensions, installed fonts, WebGL renderer, hardware concurrency, and 30+ other attributes create a fingerprint more stable than a cookie. Anti-Fingerprint mode spoofs all exposed surfaces.
Market Problem Solved
Insecure Communication
Conventional messaging routes through centralised servers, creating a single point of interception. Dhurta Connect establishes direct encrypted peer-to-peer tunnels that exist only for the session duration.
Market Problem Solved
Privacy Usability Gap
Existing privacy tools require expert configuration. Dhurta's Chakra Shield activates VPN, WebRTC blocking, anti-fingerprint, cookie guard, and ad blocking in a single interaction.

Functional Capabilities

2.1 — Sovereign Browser Core

Privacy Mode
Chakra Shield
One-tap composite privacy layer. Requires simultaneous activation of VPN, Anti-Fingerprint, and WebRTC Block. Displays animated shield indicator when all three conditions are satisfied. Real-time security breach warnings appear below the address bar if any protection layer goes offline.
Privacy Mode
Ghost Mode
In-memory ephemeral session with no disk writes, fingerprint spoofing enabled, WebRTC fully blocked, and all traffic routed through the Tor onion network. Session state is destroyed completely on deactivation.
Network Security
VPN Integration
Built-in VPN proxy rotation. Country selection for exit node. Real-time public IP verification and comparison against the device's actual IP to confirm masking is active.
Security Layer
WebRTC Leak Prevention
WebRTC STUN/ICE candidates are blocked at the Electron session layer, preventing the protocol from bypassing the VPN tunnel to expose the host device's real network interface address.
Privacy Layer
Anti-Fingerprint Engine
Spoofs screen resolution, hardware concurrency, device memory, WebGL vendor/renderer strings, navigator plugins, timezone, language, and canvas fingerprint. Each session receives a randomised but internally consistent fingerprint set.
Productivity
Omni Privacy Dashboard
Live display of current public IP, geolocation, real IP comparison, blocked request count, active fingerprint surface, and all security layer statuses. Single-screen privacy audit without leaving the browser.

2.2 — Dhurta Connect: Peer-to-Peer Secure Communications

Communication
P2P Text Chat
End-to-end encrypted real-time messaging between two peers with no message storage, no server-side logging, and no account requirement. Messages exist only in the session.
Communication
Voice & Video Calls
Direct WebRTC-based encrypted audio and video calls established through the secure P2P signalling relay. Zero third-party media servers; media streams flow peer-to-peer after handshake.
File Transfer
Secure File Sharing
File transmission occurs directly between peers over the encrypted channel. Supports share-by-direct-link and share-by-numeric-code workflows. No file data transits through any cloud intermediary.
Access Model
Numeric Code Pairing
Connections are established through short numeric session codes. No user accounts, no phone numbers, no email addresses. The pairing code is the only shared credential and expires with the session.

2.3 — Dhurta Setu: Web Index & Information Bridge

Dhurta Setu is the ecosystem's curated information gateway — a locally served web index that provides access to the open information commons without routing queries through centralised search infrastructure. Setu runs entirely within the local Dhurta context, rendering from a file-based entry point with no external API dependency. It is accessible directly from the browser's home tab Favourites dock, loading instantly as a native tool.

Client Standard Operating Procedures (SOPs)

SOP-C01 — Initialising a Secure Browsing Session

  1. 01
    Launch the Dhurta Application
    Open Dhurta from your system's application directory. The browser will initialise to the Home Tab, displaying the Trishula identity mark, a live clock, and the Favourites dock. Allow 3–5 seconds for background security services to complete initialisation.
  2. 02
    Activate Chakra Shield for Standard Privacy
    Click the Chakra icon (the spinning wheel symbol) in the left sidebar. The system will automatically connect to the VPN, enable Anti-Fingerprint, and block WebRTC. The icon will begin its active animation once all three layers confirm. If any layer fails, a security breach warning will appear below the address bar with a one-click remediation button.
    Note: Chakra Shield requires VPN connectivity. Ensure your network allows outbound VPN connections on standard ports.
  3. 03
    Activate Ghost Mode for Maximum Anonymity
    Click the Trishula logo at the top of the sidebar. Ghost Mode creates an in-memory browser session, routes all traffic through Tor, and spoofs your fingerprint. The sidebar will glow red when Ghost Mode is active. Tor routing may take 10–30 seconds to establish on first activation.
    Note: Ghost Mode and Chakra Shield are mutually exclusive — activating one will gracefully deactivate the other.
  4. 04
    Verify Your Privacy Status
    Click the Omni icon (shield with dot) in the sidebar to open the Privacy Dashboard. Confirm your displayed Public IP differs from your Real IP, that the Blocked Count is actively incrementing on pages with trackers, and that all shield layers show a green status indicator.
  5. 05
    Navigate and Browse
    Use the Search or URL bar to navigate. The address bar's security badge reflects the current page's certificate status. Use the Favourites dock on the Home Tab for quick access to Dhurta tools (Setu, Connect, Omni) and your saved sites.

SOP-C02 — Establishing a Dhurta Connect Secure Communication Session

  1. 01
    Open Dhurta Connect
    From the Home Tab Favourites dock, click the Dhurta Connect tile (linked circles icon). Alternatively, click the Tools icon in the sidebar to open the Tools popup and select Connect. The Connect interface will load within the current tab.
  2. 02
    Generate a Session Code (Host Role)
    If you are the session initiator, click Host Session. The system generates a short numeric session code. Share this code with your intended peer through any out-of-band channel (phone, in-person, another messaging platform).
    Example code format: 4829
  3. 03
    Join via Session Code (Peer Role)
    If you are the session joiner, enter the numeric code provided by the host into the Join Session field and confirm. The P2P handshake will complete, establishing a direct encrypted connection between both endpoints.
  4. 04
    Communicate and Transfer Files
    Once connected, use the chat panel for text messages, the call controls for voice and video, and the file panel to select files for direct transfer. All data flows directly between the two endpoints. No data is stored or logged by any intermediary.
  5. 05
    Terminate the Session
    Click End Session to close the P2P connection. The session code immediately becomes invalid. No session history, message logs, or transferred file metadata are retained in the application after session termination.
Client Advisory
Session codes are single-use identifiers valid only for the duration of the connection. Never re-use a session code across separate communication events. Generate a fresh code for each new session.

Volume II · Section 5

System Architecture & Cryptographic Protocols

5.1 — Process Architecture

Dhurta runs as a two-process Electron application. The Main Process (Node.js) manages all OS-level operations: BrowserWindow/BrowserView lifecycle, IPC handler registration, SQLite access via electron/db.ts, security service coordination, custom protocol registration, and child process spawning for Tor and Connect relay. The Renderer Process (React/Chromium) is isolated by contextIsolation: true and communicates exclusively through the window.dhurta API surface exposed by electron/preload.ts. No renderer code has direct access to Node.js APIs or the filesystem.

Web content is displayed within BrowserView instances — separate Chromium views attached to the main BrowserWindow. Each tab maintains its own BrowserView, allowing independent session isolation, per-tab zoom, and security policy application. Internal dhurta:// pages suppress the BrowserView entirely and render the React component tree directly, improving performance and eliminating the BrowserView stack-layer race conditions that affected navigation from the home tab.

5.2 — Zero-Cloud P2P Transport (Dhurta Connect)

Dhurta Connect achieves zero-cloud, untraceable connectivity through a two-phase architecture. In Phase 1 (Signalling), a lightweight WebSocket relay spawned locally on port 8080 brokers the initial ICE candidate exchange between two peers. This relay handles only the 4–8 signalling messages required for WebRTC handshake and carries no application data. In Phase 2 (Direct Transport), the WebRTC data channel establishes a direct peer-to-peer DTLS-SRTP encrypted connection between the two endpoints. The relay may be shut down after handshake completes; the channel persists independently.

All file data, message content, and media streams are encrypted in transit by the WebRTC DTLS-SRTP layer. No message persistence layer exists; content exists only within the live session context. Session codes are short-lived, single-use identifiers with no persistent mapping. There is no user account database, no message store, and no audit trail for any communication event.

5.3 — Security Layer Architecture (Chakra Shield)

The Chakra Shield composite state is computed as a derived boolean: ipRotation && antiFingerprint && blockWebRTC. All three conditions must be simultaneously satisfied. The state is not stored — it is recomputed from the chakraSnapshot object which stays synchronised to the underlying security settings via the dhurta:settingChanged custom event bus. This prevents stale-state divergence between the sidebar indicator and the actual security posture.

WebRTC blocking is enforced at the Electron session level, preventing the Chromium engine from exposing STUN/ICE candidates regardless of page-level JavaScript. Anti-Fingerprint spoofing is injected as a content script into every BrowserView before any page code executes, ensuring the spoofed values cannot be read by any site script during the page-load cycle.

Developer Engineering SOPs

Internal Use Only — Not for Client or Third-Party Distribution

SOP-D01 — Initial Local Stack Initialisation

  1. 01
    Verify Runtime Prerequisites
    Confirm Node.js v20 LTS or higher and pnpm v9+ are installed and available on the system PATH.
    node --version # must be >= 20.0.0 pnpm --version # must be >= 9.0.0
  2. 02
    Install Monorepo Dependencies
    Execute from the repository root. The --frozen-lockfile flag prevents unintended dependency resolution changes in the shared lockfile.
    cd P:\Project\Shiva\Dhurta pnpm install --frozen-lockfile
    Warning: Do not use npm or yarn. The pnpm workspace protocol in package.json will produce incorrect resolution with other package managers.
  3. 03
    Compile the Electron Main Process
    TypeScript compilation of the main process uses the separate tsconfig.electron.json configuration. Both configs must pass clean before the application can be started.
    npx tsc -p tsconfig.electron.json --noEmit # verify main npx tsc --noEmit # verify renderer
  4. 04
    Start Development Orchestration
    The development environment requires two concurrent processes: the Vite renderer dev server and the Electron host process. Start them in separate terminal instances in the following order.
    # Terminal 1 — Renderer pnpm dev # Terminal 2 — Electron (after Vite is listening) pnpm electron
  5. 05
    Initialise Sub-Project Dependencies (Dhurta Connect)
    Dhurta Connect maintains its own node_modules. Navigate to the sub-project root and install independently before the Electron process attempts to spawn the relay.
    cd P:\Project\Shiva\Dhurta Connect pnpm install --frozen-lockfile

SOP-D02 — Building a Production Distribution

  1. 01
    Run Full TypeScript Validation
    Both tsconfig files must emit zero errors before initiating a build. Do not proceed with a failing type check.
  2. 02
    Build Renderer Bundle
    pnpm build # outputs to dist/
  3. 03
    Package with electron-builder
    pnpm dist # produces NSIS installer in release/
    Ensure code-signing certificates are configured in electron-builder.yml before distributing externally.

7.1 — Tool Registry Path Configuration

Each tool's absolute path is declared in electron/tools/{toolName}.ts as the projectRoot field of the ToolEntry interface. Static-type tools (Setu) specify an entryFile resolved relative to projectRoot, loaded as a file:// URL. Server-type tools (Connect) specify relayEntry, relayPort, and clientPort; the relay is spawned as a child process and the client SPA is served by an in-process Node HTTP server.

The resolveToolUrl(id) function in electron/tools/index.ts is the single resolution point for all dhurta-tool:// protocol navigation. It handles both tool types and returns a Promise<string | null> — null indicates the tool could not be started and navigation should be aborted.

Attack Surface Assessment & Risk Ledger

Classification — Restricted
This section documents known vulnerabilities, their mitigations, and residual risk. Do not share outside the engineering team. Do not commit this section unredacted to any repository accessible beyond the core team.
# Attack Vector / Risk Severity Surface Mitigation Implemented Residual Risk
R-01 WebRTC IP Leak — WebRTC STUN/ICE negotiation exposes host device real IP and LAN IP, bypassing VPN tunnel at the OS network layer. Critical Browser engine (Chromium) WebRTC blocked at Electron session level when blockWebRTC is true. Applied before any page code executes. Real-time breach warning banner displayed if blockWebRTC goes false while browsing. Low — Residual only if user explicitly disables the setting or if a future Chromium update changes the session API surface.
R-02 Browser Fingerprint Persistence — Stable fingerprint across sessions (screen, WebGL, canvas, navigator) allows cross-site re-identification without cookies or local storage. Critical Renderer, BrowserView Anti-Fingerprint content script injected before page load when enabled. Randomises screen dimensions, hardware concurrency, device memory, WebGL strings, navigator plugins, and canvas noise per session. Breach banner triggers if setting is off. Medium — Font enumeration and timing-based fingerprinting remain partially exploitable by sophisticated adversaries. Full mitigation requires Ghost Mode.
R-03 VPN Split-Tunnel / Connection Failure — VPN proxy fails to connect (no provider configured, network block), leaving IP unmasked while user believes they are protected. Critical Network layer, IPC Chakra Shield requires confirmed VPN connection (vpnConnect() success) before activating. Shield icon remains idle on failure. SecurityBreachBanner displays an IP-exposed warning with one-click fix. Chakra state strictly gated on ipRotation && antiFingerprint && blockWebRTC. Medium — Depends on external VPN provider reliability. Tor-backed Ghost Mode eliminates this residual.
R-04 Windows Child-Process Spawn Failure — Node.js child_process.spawn() on Windows fails silently or with ENOENT when the target script path contains spaces, uses an incorrect working directory, or when the Node.js runtime is not on the system PATH in the Electron context. High Electron main process, tool relay spawn Relay spawn in electron/tools/index.ts uses explicit cwd option set to the tool's projectRoot. All paths are passed as absolute strings. Runtime is resolved to the Electron-bundled Node binary, not the system PATH. Spawn errors are caught and logged to the main process console; resolveToolUrl() returns null on failure to prevent navigation hang. Low — Known edge case on paths containing Unicode characters or exceeding 260 chars (MAX_PATH). Mitigated by the P:\Project\Shiva\ path convention which stays within limits.
R-05 BrowserView Visibility Race — Navigating to a dhurta-tool:// URL from the home tab (where all BrowserViews are removed) loads content into a hidden BrowserView, resulting in blank tab display with active background load. High Electron BrowserView lifecycle, IPC nav handler The nav:loadURL IPC handler explicitly sequences: remove all BrowserViews → addBrowserView(tab.view) → setBounds() → setAutoResize() → loadURL() before initiating the load. The sequence is atomic within the IPC call. Low — Fixed. Regression risk if the nav handler is modified without preserving the show-before-load sequence.
R-06 IPC Channel Accumulation — Repeated tab switches without correctly removing ipcRenderer.on() listeners cause duplicate event handlers, resulting in N copies of callbacks firing (e.g., N new tabs opened per window.open() call). High Renderer preload, IPC listener registry Preload implements a _wrappers Map tracking the anonymous wrapper function created by each .on() call. The .off() implementation retrieves and removes the exact wrapper. Guard on .on() de-registers any existing wrapper for the same channel+callback before adding a new one, preventing accumulation even if off() is skipped. Low — Fixed. Only re-emerges if a new channel is added to VALID_CHANNELS without following the established wrapper pattern.
R-07 P2P Relay Port Conflict — Dhurta Connect relay binds to port 8080 on localhost. If another process occupies this port, relay spawn silently fails or crashes, leaving the user unable to establish Connect sessions. Medium Dhurta Connect, tool relay Runtime state map in electron/tools/index.ts tracks relay process health. ensureServerTool() checks whether the relay is already running before spawning a second instance. On before-quit, shutdownAllTools() terminates all relay processes and closes HTTP servers cleanly. Medium — No automatic port fallback implemented. Manual conflict resolution required. Future: dynamic port allocation with relay advertising the selected port to the client.
R-08 Chakra State Divergence — Security settings modified from multiple surfaces (sidebar, Omni page, Security panel) could produce inconsistent displayed state if each surface maintains independent local state not synchronised to a single source of truth. Medium State management (useBrowser.ts, OmniPage) All security setting mutations dispatch dhurta:settingChanged custom events. All consumers (sidebar store, Omni page, App.tsx securityStatus) subscribe to the same event bus and update their local derived state reactively. chakraActive is never stored; it is derived on every render from the live snapshot. Low — Eliminated by the event-bus pattern. Residual risk only if a new settings surface is added without dispatching the event.
R-09 Contextual Isolation Bypass Risk — Malicious page JavaScript attempting to access Electron internals or Node.js APIs through the renderer process. Critical Electron security model contextIsolation: true enforced on all BrowserView instances. nodeIntegration: false. Only the explicit window.dhurta API surface is exposed via contextBridge. No arbitrary IPC channels are exposed. All handler inputs are validated type-narrowed before use. Low — Standard Electron security best practices followed. Risk increases only if contextIsolation is ever disabled for debugging and not re-enabled.
